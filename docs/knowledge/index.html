ディレクトリ構成

NestJSの機能モジュールをどこに置くか、どう分けるか。

NestJSではどのようなディレクトリ構成パターンがある？

NestJSコミュニティで見かける代表的なパターンは3つ。「公式の絶対ルール」は存在せず、規模・チームの好みで選ぶ。

A. フラット派（公式CLIのデフォルト）

nest g resource categories を実行すると src/categories/ 直下にフラットに作られる。

src/ ├─ categories/ │ ├─ categories.module.ts │ ├─ categories.controller.ts │ └─ ... ├─ transactions/ ├─ users/ ├─ prisma/ ├─ common/ └─ app.module.ts
  • ○ 公式CLIをそのまま使える、階層が浅い
  • × 機能が10〜20個に増えると src/ 直下に機能・横断的処理・設定が混在してカオスになりがち

B. modules配下派採用

機能モジュールを src/modules/ 配下にまとめ、横断的処理（common, prisma）と物理的に分離する。

src/ ├─ modules/ │ ├─ categories/ │ └─ transactions/ ├─ common/ ├─ prisma/ └─ app.module.ts
  • ○ 「機能モジュール」と「横断的処理」が物理的に分離されてスッキリ
  • ○ 機能が増えても modules/ に並ぶだけで src/ 直下が荒れない
  • × 階層が1つ深くなる、公式CLIだと --path src/modules 指定が必要

C. DDD（ドメイン駆動設計）

「ドメイン（業務ルール）を中心に据える」設計思想を取り入れたパターン。後述の専用Q参照。

業界での採用率（体感）

パターン採用率適性規模
A. フラット多い（公式デフォルト）小〜中規模
B. modules配下そこそこ多い中〜大規模、機能が増える前提
C. DDD少ない大規模・複雑ドメイン
pick-cashの判断: 機能が Category / Transaction / 将来の Income / Report ... と増えていく前提なので B（modules配下派） を採用。今のうちに modules で分けておくと、後から物理的な引っ越しが不要になる。
pick-cashの最終的なディレクトリ構成はどうなる？

common/（横断的）と modules/（縦割り）を分けた構成。各機能モジュールは「Controller / Service / Repository / DTO / Module」の5層で構成する。

apps/backend/src/ ├─ common/ ← 横断的関心事 │ ├─ filters/ │ │ ├─ globalException.filter.ts │ │ └─ globalException.filter.spec.ts │ └─ middlewares/ │ ├─ auth.middleware.ts │ └─ auth.middleware.spec.ts ├─ modules/ ← 機能モジュールはここに集約 │ └─ categories/ │ ├─ categories.module.ts ← DI設定 │ ├─ categories.controller.ts ← HTTPエンドポイント定義 │ ├─ categories.service.ts ← ビジネスロジック層 │ ├─ categories.repository.ts ← Prisma操作集約・論理削除フィルタ │ ├─ categories.service.spec.ts │ ├─ categories.repository.spec.ts │ └─ dto/ │ └─ getCategories.dto.ts ← クエリパラメータ用DTO ├─ prisma/ ← PrismaService ├─ types/ ← 型拡張（.d.ts） ├─ generated/ ← Prisma生成物 ├─ app.controller.ts ├─ app.module.ts ← CategoriesModule の imports に追加 └─ main.ts ← ZodValidationPipe をグローバル登録

各ディレクトリの役割

ディレクトリ役割入れるもの
common/アプリ全体に効く横断的処理Filter / Middleware / Pipe / Guard
modules/機能単位のモジュール各機能のサブディレクトリ（categories, transactions, users…）
prisma/DB接続PrismaService
types/型拡張.d.ts 系
各機能モジュール（5層）の役割と依存関係は？

各機能モジュールは「Controller → Service → Repository → Prisma」と下層を呼び出す依存関係で構成される。各層が単一の役割を持つ。

依存方向

HTTPリクエスト ↓ Controller ← HTTPエンドポイント定義、リクエスト/レスポンス変換だけ ↓ 依存 Service ← ビジネスロジック層 ↓ 依存 Repository ← Prismaへのアクセス集約、deletedAt IS NULL を強制 ↓ 依存 PrismaClient

各層の責務

層役割NGなこと
ControllerHTTPエンドポイント定義、リクエスト/レスポンス変換ビジネスロジックを書く、Prismaを直接呼ぶ
Serviceビジネスロジック、複数Repositoryをまたいだ処理Prismaを直接呼ぶ
RepositoryPrisma操作の集約、論理削除フィルタの強制HTTPの知識を持つ、ビジネスロジックを持つ
DTO入出力の型と契約ロジックを持つ
ModuleDIの配線ロジックを持つ

実装する順番

2つの流派がある。

  1. トップダウン: Controller → Service → Repository（APIの形から逆算）
  2. ボトムアップ: Repository → Service → Controller（下層から積み上げる）推奨

ボトムアップの方が、書いてる途中に型補完が効くので推奨。pick-cashでは DTO → Repository → Service → Controller → Module → AppModule の順で書く。

DTOを先に書くのは、入出力の「型契約」を最初に決められるから。transactionTypeSchema のような shared 部品を使い回せる。
DDDパターンを採用するとどんな構成になる？

DDD（Domain-Driven Design）は「ドメイン（業務ルール）を中心に据える」設計思想。4つのレイヤーで構成する。

DDDの4レイヤー

┌────────────────────────────────┐ │ Presentation（外との接点） │ ← Controller、リクエスト/レスポンス変換 ├────────────────────────────────┤ │ Application（ユースケース） │ ← Service的役割、トランザクション境界 ├────────────────────────────────┤ │ Domain（業務ルール） │ ← エンティティ、値オブジェクト、Repositoryの契約 ├────────────────────────────────┤ │ Infrastructure（外部依存実装） │ ← Repository実装、Prisma、外部API └────────────────────────────────┘ ↑ 依存方向（外側→内側のみ）

ポイント: Domain層は 外部（Prisma等）に依存しない。Repository は Domain では interface だけ定義して、実装は Infrastructure に置く（依存性逆転）。

スタイル1: レイヤー先頭（水平分割）

レイヤーを src/ 直下に並べる。

src/ ├─ presentation/ │ └─ categories/ │ ├─ categories.controller.ts │ └─ dto/getCategories.dto.ts ├─ application/ │ └─ categories/ │ └─ getCategoriesUsecase.ts ├─ domain/ │ └─ categories/ │ ├─ category.entity.ts │ ├─ transactionType.vo.ts ← 値オブジェクト │ └─ categoryRepository.interface.ts ← Repository契約 └─ infrastructure/ └─ categories/ └─ prismaCategoryRepository.ts ← Repository実装

スタイル2: コンテキスト先頭（垂直分割）

機能（境界づけられたコンテキスト）を先頭に置いて、その下にレイヤー。

src/ └─ contexts/ ├─ categories/ │ ├─ presentation/ │ ├─ application/ │ ├─ domain/ │ ├─ infrastructure/ │ └─ categories.module.ts └─ transactions/ ├─ presentation/ ├─ application/ ├─ domain/ ├─ infrastructure/ └─ transactions.module.ts

コード例（スタイル1）

// domain/categories/categoryRepository.interface.ts
export interface CategoryRepository {
  findAll(type?: TransactionType): Promise<Category[]>;
}

// infrastructure/categories/prismaCategoryRepository.ts
@Injectable()
export class PrismaCategoryRepository implements CategoryRepository {
  constructor(private prisma: PrismaService) {}
  async findAll(type?: TransactionType): Promise<Category[]> { ... }
}

// application/categories/getCategoriesUsecase.ts
@Injectable()
export class GetCategoriesUsecase {
  constructor(
    @Inject('CategoryRepository') private repo: CategoryRepository
  ) {}
  async execute(type?: TransactionType) { return this.repo.findAll(type); }
}

どちらが「DDDらしい」？

スタイル2（コンテキスト先頭）の方がDDDの「境界づけられたコンテキスト」という概念に近い。書籍『実践ドメイン駆動設計』（IDDD）でもこちらが推奨。NestJSの @Module() がそのまま境界づけられたコンテキストになるので相性が良い。

pick-cashにDDDを採用しなかった理由

今回のpick-cashDDDが効くプロジェクト
ドメインの複雑さ単純（取引のCRUD）業務ルールが複雑（医療・金融・物流）
規模個人開発・小〜中規模大規模・複数チーム
長期保守学習プロジェクト数年〜10年単位

DDDの interface分離、依存性逆転、値オブジェクト あたりは、業務ルールが薄いプロジェクトに入れると「コードは増えるけど価値は出ない」ことが多い。学習目的で 「将来やるかもしれないからイメージは持っとく」 のはアリ。

@デコレータ

NestJSがクラス・メソッドに「メタデータをくっつける」仕組み。NestJS理解の核。

そもそも@デコレータって何？何のためにある？

NestJSのデコレータは 「クラスやメソッドに『メタデータ』をくっつける」 仕組み。NestJSがクラスを見たときに、デコレータを読んで「あ、これはこういう用途なのね」と理解する。

デコレータが付いてないクラス vs 付いてるクラス

// ただのクラス。NestJSは何も特別扱いしない
class Foo {}

// @Module() が付いた瞬間、NestJSは「これはModuleだ」と認識する
@Module({})
class Bar {}

デコレータの主な役割

役割例
クラスの種別を宣言@Module(), @Controller(), @Injectable()
HTTPメソッドとパスを紐付け@Get(), @Post(), @Patch()
リクエストの要素を取り出す@Body(), @Query(), @Param()
横断的処理を適用@UseGuards(), @UseFilters(), @UsePipes()
例外の対象を指定@Catch()
イメージ: デコレータは「クラスに貼る付箋」みたいなもの。NestJSが起動時にクラスを見て、付箋を読んで「Moduleとして登録するぞ」「Controllerとして扱うぞ」と判断する。
クラスレベルのデコレータ（@Module / @Injectable / @Controller / @Catch）の違いは？

「このクラスは何者か」を宣言する系のデコレータ。種類ごとに役割が違う。

@Module()

「このクラスは DIコンテナに登録するモジュールの単位 です」と宣言。imports / providers / controllers / exports を引数で渡す。

@Module({
  imports: [PrismaModule],
  controllers: [CategoriesController],
  providers: [CategoriesService, CategoriesRepository],
})
export class CategoriesModule {}

@Injectable()

「このクラスは DIコンテナの管理対象 です」と宣言。Service / Repository / Guard / Interceptor / Pipe / Middleware など、DIで他クラスから注入される側のクラスにつける。

@Injectable()
export class CategoriesService {
  constructor(private readonly repo: CategoriesRepository) {}
  //                  ↑ DIコンテナが自動で渡してくれる
}

@Controller()

「このクラスは HTTPエンドポイントを担当するController です」と宣言。引数でルートプレフィックスを指定。

@Controller('categories')   // ← /categories がルートプレフィックス
export class CategoriesController {
  @Get()                    // ← GET /categories
  async list() { ... }

  @Get(':id')               // ← GET /categories/:id
  async detail() { ... }
}

@Catch()

「このクラスは ExceptionFilter で、これこれの例外を担当します」と宣言。

@Catch()                              // 引数なし: 全例外を担当
@Catch(HttpException)                 // 引数あり: HttpExceptionとその子孫だけ
@Catch(HttpException, ZodValidationException)   // 複数指定もOK

@Catch() が付いてないクラスは、useGlobalFilters() に渡しても ExceptionFilter として認識されない。

使い分けの表

デコレータ付けるクラス意味
@Module()機能の束ね役クラスDIコンテナに登録する単位
@Injectable()Service, Repository, Middleware, Guard, Interceptor, PipeDI管理対象
@Controller()ControllerHTTPエンドポイント担当
@Catch()ExceptionFilter例外捕捉担当、対象例外を指定
@Injectable() は広い概念（DI管理対象なら何でも）。@Controller() や @Catch() は狭い具体的な役割を宣言する。
メソッドレベルのデコレータ（@Get / @Post / @Body / @Query）は何をしている？

Controllerの 各メソッドにHTTPメソッドとパスを紐付けたり、リクエストから値を取り出したり するためのデコレータ。

HTTPメソッド系

@Controller('categories')
export class CategoriesController {
  @Get()           // GET /categories
  list() {}

  @Get(':id')      // GET /categories/:id
  detail() {}

  @Post()          // POST /categories
  create() {}

  @Patch(':id')    // PATCH /categories/:id
  update() {}

  @Delete(':id')   // DELETE /categories/:id
  remove() {}
}

リクエスト要素を取り出す系

デコレータ取り出すもの使用例
@Body()リクエストボディ全体POST /categories のJSON
@Query()クエリパラメータ全体?type=expense
@Param()URLパスパラメータ/categories/:id の id
@Req()Expressのreq全体req.userId など
@Res()Expressのres全体通常使わない（NestJSの自動整形が無効になる）
 
@Get(':id')
async detail(
  @Param('id') id: string,            // /categories/:id の id だけ取り出す
  @Query() query: GetQueryDto,        // クエリ全体をDTOで受ける
  @Body() body: CreateBodyDto,        // ボディ全体をDTOで受ける
  @Req() req: Request,                // Expressのreqを直接触る
) {
  return { userId: req.userId, id };  // AuthMiddlewareが入れたuserIdが取れる
}
@Body() / @Query() / @Param() にDTO型を組み合わせると、自動でバリデーションが走る（ZodValidationPipeが効いている場合）。
横断的処理を適用するデコレータ（@UseGuards / @UseFilters / @UsePipes / @UseInterceptors）は？

NestJSには「横断的関心事」を扱う仕組みが4種類ある。それぞれ Guard / Filter / Pipe / Interceptor と呼ばれ、@UseXxx() デコレータで特定のController/メソッドに適用できる。

4種類の役割

種類役割適用デコレータ
Guard認可。リクエストを通すか拒否するか判定@UseGuards()
Interceptor前後処理（ロギング、キャッシュ、レスポンス変換）@UseInterceptors()
Pipe入力の変換・バリデーション@UsePipes()
ExceptionFilter例外処理@UseFilters()

適用範囲の3パターン

  1. 特定のメソッド/Controllerだけ: @UseXxx() をクラスやメソッドに付ける
  2. アプリ全体（main.ts）: app.useGlobalXxx(new ...)
  3. アプリ全体（providers経由でDI使う）: { provide: APP_GUARD, useClass: ... }

例: 特定のメソッドだけ

@Controller('transactions')
@UseGuards(AuthGuard)                       // Controller全体に
export class TransactionsController {
  @Get(':id')
  @UseInterceptors(LoggingInterceptor)      // このハンドラだけ
  findOne(@Param('id', ParseIntPipe) id: number) { ... }
  //                       ↑ パラメータレベルでPipe
}

例: アプリ全体（main.ts）

const app = await NestFactory.create(AppModule);
app.useGlobalFilters(new GlobalExceptionFilter());
app.useGlobalPipes(new ZodValidationPipe());

例: アプリ全体（DI効かせる場合）

@Module({
  providers: [
    { provide: APP_GUARD, useClass: AuthGuard },
    { provide: APP_FILTER, useClass: ExceptionFilter },
  ],
})
export class AppModule {}

Cの利点は DIコンテナ経由でインスタンス化されるので、他のService（PrismaServiceなど）を注入できること。Bは new XxxFilter() を自分でやるので、依存があると面倒。

Middlewareだけ@UseXxx()がないのはなぜ？

NestJSの横断的処理 5種類のうち、Middlewareだけ登録方法が違う。@UseMiddleware() は存在せず、代わりに AppModule.configure() で登録する。

Middlewareの登録方法

@Module({...})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer): void {
    consumer.apply(AuthMiddleware).forRoutes('*');
  }
}

なぜMiddlewareだけ特別？

MiddlewareはExpress（またはFastify）の 生のreq/resを扱うレイヤーで、NestJSの抽象化（DIや@UseXxxの仕組み）より一段下のレイヤーに位置するから。だからNestJSは「configure() という専用フック」を Module に用意した。

登録方法のまとめ

種類場所理由
MiddlewareAppModule.configure()Expressレイヤーの特殊扱い
Guard / Interceptor / Pipe / Filtermain.ts か @UseXxx() か providers: [APP_*]NestJSレイヤーで色々な粒度に適用したいから選択肢が多い

NestJSのリクエスト処理パイプライン

Request ↓ Middleware ← AppModule.configure() で登録 ↓ Guard ← @UseGuards() / グローバル登録 ↓ Interceptor ← @UseInterceptors() / グローバル登録 ↓ Pipe ← @UsePipes() / グローバル登録 ↓ Controller ← @Controller(), @Get() などで定義 ↓ Service / Repository ↓ 例外発生 ← ExceptionFilter ← @UseFilters() / グローバル登録
覚え方: 「Moduleクラスに implements 何か を付けてその中で前処理を書く」のは Middlewareだけの特殊パターン。
@Module()の引数（imports / providers / controllers / exports）の意味は？

@Module() は4つの主要オプションを受け取る。NestJSのDI設定の中心。

各オプションの役割

キー意味何を入れる
imports他Moduleから借りてくる他のModuleクラス
controllersこのModuleが持つControllerControllerクラス
providersこのModuleで生成・管理するクラスService / Repository など
exports他Moduleにも公開するクラスprovider群の一部

使用例

// PrismaModule: PrismaServiceを他Moduleで使えるようexportする
@Module({
  providers: [PrismaService],
  exports: [PrismaService],   // ← exportしないと他Moduleから使えない
})
export class PrismaModule {}

// CategoriesModule: PrismaModuleからPrismaServiceを借りる
@Module({
  imports: [PrismaModule],    // ← importすればPrismaServiceが使える
  controllers: [CategoriesController],
  providers: [CategoriesService, CategoriesRepository],
})
export class CategoriesModule {}

DIコンテナの仕組み

これらを @Module({...}) で宣言することで、NestJSのDIコンテナに「何を登録するか」を伝える。

NestJSアプリ └─ DIコンテナ（NestJSが裏で持つ「箱」） ├─ AppModule │ ├─ AppController │ └─ AuthMiddleware └─ PrismaModule └─ PrismaService
AppModule ≠ DIコンテナ。 AppModuleは「DIコンテナに何を登録するかをまとめた設計図」みたいなもの。コンテナ本体はNestJSの内部にある。
implementsとデコレータはどう違う？両方ついてるクラスがあるけど…

2つは全く別の役割を持つ。デコレータは「メタデータ付与」、implementsは「型契約の宣言」。両方付くケースは普通にある。

例: AuthMiddleware

@Injectable()                                      // ← デコレータ
export class AuthMiddleware implements NestMiddleware {  // ← implements
  use(req, res, next) { ... }
}
@Injectable()（デコレータ）implements NestMiddleware
役割NestJSに「DI管理対象」とメタデータ付与「use メソッドを持つ契約」をTypeScriptに宣言
影響範囲ランタイム（NestJS）コンパイル時（TypeScript）
なくしたら？NestJSがDI管理してくれない型エラーが消えるが、書き間違い検知ができなくなる

implementsとextendsの違い

extends（継承）implements（契約）
意味親から実装を引き継ぐ形だけ満たすことを約束
実装は？親から自動で引き継ぐ自分で全部書く
多重指定1個だけ複数OK（implements A, B, C）
用途コード再利用「この形を保証」

3者が組み合わさった例

@Injectable()
export class PrismaService
  extends PrismaClient                    // ← PrismaClientを継承（実装も引き継ぐ）
  implements OnModuleInit, OnModuleDestroy {  // ← 2つの契約を満たす
}
  • @Injectable(): 「DI管理対象」
  • extends PrismaClient: PrismaClientの機能を継承
  • implements OnModuleInit, OnModuleDestroy: ライフサイクルフックの契約を満たす
覚え方: デコレータは「NestJSへの付箋」、implementsは「TypeScriptへの宣言」、extendsは「親の実装をもらう」。影響するレイヤーが違うのがポイント。
constructor(private readonly service: CategoriesService) {} の構造を分解すると？

NestJSのServiceやControllerでよく出てくるこの書き方は、TypeScriptの 「コンストラクタパラメータプロパティ」 という省略記法。3つの要素に分解できる。

3つの要素

要素意味
private readonlyコンストラクタパラメータプロパティ。クラスフィールド宣言と this.xxx = xxx 代入を同時にやる省略記法
service変数名（クラスフィールド名）。this.service でアクセスする
: CategoriesService型注釈。ただの型ではなく DIキー として使われる

省略しない書き方 vs 省略形

// 省略しない書き方（普通のTypeScript）
@Injectable()
export class CategoriesController {
  private readonly service: CategoriesService;

  constructor(service: CategoriesService) {
    this.service = service;
  }
}

// 省略形（コンストラクタパラメータプロパティ）
@Injectable()
export class CategoriesController {
  constructor(private readonly service: CategoriesService) {}
  //          ↑ フィールド宣言 + this代入が自動化される
}

: CategoriesService はただの型ではなくDIキー

TypeScriptの型は通常コンパイル後に消えるが、tsconfig で emitDecoratorMetadata: true を有効にすると、デコレータが付いたクラスの引数型情報だけランタイムに残る。NestJSはこの型情報を reflect-metadata で読み取り、「この引数には CategoriesService のインスタンスを渡せばいい」と判断する。

コンパイル後の擬似コード
// 書いたコード
@Injectable()
class Foo {
  constructor(private service: CategoriesService) {}
}

// コンパイル後（概念）
Reflect.defineMetadata('design:paramtypes', [CategoriesService], Foo);
//                                          ↑ ここに型情報が残る

NestJSは起動時に各クラスに対して Reflect.getMetadata('design:paramtypes', クラス) を呼んで、必要な依存クラスを把握 → DIコンテナから取り出して注入する、という流れになる。

tsconfig の有無で何が変わる？

項目OFFのときONのとき
書き方同じ同じ
型情報の寿命コンパイル時に消えるランタイムに残る

注意: NestJSのDIを成立させるには tsconfig に experimentalDecorators: true と emitDecoratorMetadata: true の 両方 が必要。@nestjs/cli で初期化するとデフォルトで入っている。
@Get() に対応する list メソッドは誰が呼んでる？

ユーザーが書いたコード上では list() はどこからも呼ばれていない。NestJSが起動時にExpressへルートを登録し、HTTPリクエスト時にExpress経由で呼び出される。

メソッド名は自由

@Controller('categories')
export class CategoriesController {
  @Get()  list() {}        // OK
  @Get()  findAll() {}     // OK
  @Get()  hogeFuga() {}    // OK（動く）
}

ルーティングを決めているのは デコレータ。メソッド名は人間が読むためだけのもの。

NestJSが起動時にやっていること（擬似コード）

for (const controller of allControllers) {
  const prefix = Reflect.getMetadata('path', controller);
  for (const method of methods(controller)) {
    const httpMethod = Reflect.getMetadata('method', method);
    expressApp[httpMethod](`/${prefix}/...`, async (req, res) => {
      const instance = diContainer.get(controller);
      const result = await instance[methodName](...);
      res.json(result);
    });
  }
}

つまり起動時に「URLとメソッドの紐付け」をExpressに全部教え込んでいる。

HTTPリクエスト時の流れ

クライアント │ GET /categories ▼ Express （NestJSが裏で立てているHTTPサーバ） │ 起動時に登録されたルートにマッチ ▼ NestJSのルーティング層 │ DIコンテナからCategoriesControllerを取り出す ▼ controller.list() ← ここで初めて呼ばれる │ ▼ Service → Repository → Prisma

HonoとNestJSの違い

HonoNestJS
ルート登録自分で app.get('/categories', handler) と書くデコレータ @Get() を貼るとフレームワークが裏で登録
関数の呼び出し元自分が紐付けたhandlerが呼ばれるNestJSが自動で対象メソッドを呼ぶ
スタイル明示的（書いた通りに動く）規約型（デコレータの意味を知っている前提）
覚え方: Honoは「自分で配線する」、NestJSは「フレームワークが配線を読み取る」。NestJSは規約と引き換えに記述量が減る代わりに、「なぜ動くのか」が裏側にある。
@Controller('categories') の引数の意味は？

@Controller() の引数は ルートのプレフィックス。そのControllerに属するすべてのメソッドのパスの先頭につく。@Get() / @Post() などの引数と組み合わせて最終パスが決まる。

組み合わせ例

@Controller('categories')
export class CategoriesController {
  @Get()             // → GET    /categories
  list() {}

  @Get(':id')        // → GET    /categories/:id
  detail() {}

  @Post()            // → POST   /categories
  create() {}

  @Patch(':id')      // → PATCH  /categories/:id
  update() {}
}

プレフィックスを変えるとどうなる？

@Controller('api/v1/categories')
export class CategoriesController {
  @Get()             // → GET /api/v1/categories
  @Get(':id')        // → GET /api/v1/categories/:id
}
覚え方: 最終パス ＝ @Controller(プレフィックス) + @Get(サブパス)。プレフィックスを変えるだけで一括して全エンドポイントのパスを差し替えられる。
@Module() の providers と imports と @Global() の関係は？

3つはすべて「DI（依存性注入）で何が使えるか」をコントロールする仕組み。それぞれ違うレイヤーで効く。

3つの違い

要素役割
providersそのModule内でDI可能なクラスを登録する。同Module内のクラス同士は自由に注入し合える
imports他Moduleが exports しているproviderを借りる。importしないと外のproviderは注入できない
exports他Moduleから借りられるように公開する。providers に登録したクラスを exports にも書く必要がある
@Global()そのModuleの exports を アプリ全体で利用可能にする。importsへの記述が不要になる

providers の意味（同Module内で相互注入）

@Module({
  providers: [CategoriesService, CategoriesRepository],
})
export class CategoriesModule {}

// CategoriesService は CategoriesRepository を注入できる
// （同じModuleの providers に並んでいるので）
@Injectable()
export class CategoriesService {
  constructor(private readonly repo: CategoriesRepository) {}
}

imports + exports の関係

// 公開側: exports しないと外から使えない
@Module({
  providers: [PrismaService],
  exports: [PrismaService],   // ← これがないと借りられない
})
export class PrismaModule {}

// 借りる側: imports に書く
@Module({
  imports: [PrismaModule],    // ← これで PrismaService が注入可能に
  providers: [CategoriesService],
})
export class CategoriesModule {}

@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})

// 他Moduleでは imports に PrismaModule を書かなくてOK
@Module({
  providers: [CategoriesService, CategoriesRepository],
  // imports: [PrismaModule] ← 不要！
})
export class CategoriesModule {}

pick-cashでは PrismaModule が @Global() なので、各機能Moduleで imports: [PrismaModule] を書かなくても PrismaService が注入できる。

4要素の比較表

providersimportsexports@Global()

書く場所@Module({...})@Module({...})@Module({...})クラスの上
これだけ使う？必須必要なときだけ外に貸す場合のみ頻繁に使われるModule（Prisma, Configなど）に限定
注意: @Global() を乱用すると依存関係が不透明になる。「全Moduleが使うことが明らかなもの（DB接続、設定）」だけに限定するのが定石。

NestJSの内部の仕組み

NestJSがどうやってDI・ルーティング・バリデーションを成立させているか、その裏側。

DIコンテナってどんな仕組み？

DIコンテナは 「型をキーにインスタンスを保管する箱」 というメンタルモデルで理解するのが一番早い。NestJSが起動時にこの箱を組み立てて、必要な場所にインスタンスを渡してくれる。

DIコンテナの中身（概念図）

┌─────────────────────────────────────┐ │ DIコンテナ │ │ ┌─────────────────────────────┐ │ │ │ Key: CategoriesService │ │ │ │ Value: 起動時生成のインスタンス │ │ │ └─────────────────────────────┘ │ │ ┌─────────────────────────────┐ │ │ │ Key: CategoriesRepository │ │ │ │ Value: インスタンス │ │ │ └─────────────────────────────┘ │ │ ┌─────────────────────────────┐ │ │ │ Key: PrismaService │ │ │ │ Value: インスタンス │ │ │ └─────────────────────────────┘ │ └─────────────────────────────────────┘

「型 = 住所」「import vs 型注釈」の役割分離

importしているからDIされる…のではない。importとDIは 役割が完全に分かれている。

import文型注釈（: CategoriesService）
誰が使うTypeScript（コンパイラ）NestJS（DIコンテナ）
役割「このシンボルどこにある？」の解決「DIコンテナのどのキーを引くか」のキー指定
消えるタイミングコンパイル後にrequireに変換通常は消えるが emitDecoratorMetadata でランタイムに残る

つまり、ファイルパスやimport文はTypeScriptに「シンボルの場所」を教えるためのもので、DIコンテナはそれを 気にしない。DIコンテナは「型」だけを見てインスタンスを引っ張ってくる。

デフォルトはシングルトン

NestJSのDIは 起動時に1回だけインスタンスを生成し、使い回すのがデフォルト。100リクエスト来ても同じ CategoriesService インスタンスが使われる。scope: Scope.REQUEST を指定するとリクエストごとに生成されるが、特別な事情がない限り使わない。

覚え方: 「DIコンテナはMap<型, インスタンス>」。起動時にこのMapを組み立てて、コンストラクタの型を見て該当インスタンスを取り出して渡す。これがNestJSのDIの本体。
interfaceでDIしたい場合の特別な書き方は？

通常の constructor(private repo: CategoryRepository) {} は クラス型でしか動かない。interface型でDIしたい場合は @Inject('TOKEN') でトークンを明示する必要がある。

なぜinterfaceは普通に書けないのか

interfaceは コンパイル時に消える。emitDecoratorMetadata でメタデータを残しても、interfaceは「型情報」として存在しないため、Reflectで取り出すと Object になってしまう。NestJSは「Object型？知らないキーだな」となってDIに失敗する。

コード例

// ❌ 動かない（interfaceはランタイムに型情報が残らない）
@Injectable()
export class GetCategoriesUsecase {
  constructor(private readonly repo: CategoryRepository) {}
  //                                ↑ コンパイル後は Object になる
}

// ✅ @Inject() でトークンを明示
@Injectable()
export class GetCategoriesUsecase {
  constructor(
    @Inject('CategoryRepository') private readonly repo: CategoryRepository
  ) {}
}

// providers 側でも文字列トークンで紐付け
@Module({
  providers: [
    { provide: 'CategoryRepository', useClass: PrismaCategoryRepository },
  ],
})
export class CategoriesModule {}

いつ出てくるか

  • DDDで Repository を interface で分離する場合
  • テスト用にモックと本番実装を切り替えるアーキテクチャ
  • 外部API クライアントを抽象化する場合
pick-cashでは登場しない: Repositoryはinterface分離せず具体クラスを直接注入する設計。よってこの @Inject() パターンは出てこない。「DDDをやるなら必須の書き方」として覚えておくレベルでOK。
リクエストが届いてからControllerメソッドが呼ばれるまでの全体像は？

NestJSの処理パイプラインは Middleware → Guard → Interceptor → Pipe → Controller → Service → Repository。例外が出たらExceptionFilterに合流する。

リクエスト処理パイプライン

Request ↓ Middleware ← AuthMiddleware で req.userId をセット ↓ Guard ← 認可チェック（pick-cashでは未使用） ↓ Interceptor ← 前後処理（ロギング等。pick-cashでは未使用） ↓ Pipe ← ZodValidationPipe がDTOを検証 ↓ Controller ← @Get() で紐付いたメソッドが呼ばれる ↓ Service ← ビジネスロジック ↓ Repository ← Prismaを叩く ↓ DB [例外が出たら] ← GlobalExceptionFilter に合流、ステータスコードとメッセージを整形して返す

ZodValidationPipeが効く位置

Pipeレイヤーは「Controller直前」。@Body() dto: CreateCategoryDto のような書き方をすると、Pipeが DTOに紐付いたZodスキーマで検証 → 失敗ならZodValidationException。検証が通った値だけControllerに渡る。

起動時 vs リクエスト時 の対比

起動時に1回だけリクエストのたびに毎回
1tsconfigの emitDecoratorMetadata で生成されたメタデータを reflect-metadata 経由で読むExpress がリクエストを受ける
2DIコンテナにシングルトンを生成・登録Middleware → Guard → Interceptor → Pipeの順で実行
3Expressに全ルートを登録DIコンテナからController取り出し → メソッド実行
4Middlewareを AppModule.configure() で登録例外発生時は ExceptionFilter へ
覚え方: 起動時にやることは「メタデータ読込 + DIコンテナ構築 + ルート登録」。リクエスト時は「パイプラインを順に通すだけ」。重い仕事は起動時に終わっているのがNestJSの設計思想。

API実装の進め方

機能モジュール（Category APIなど）を実装する際の標準的なフローと各層の責務。

機能APIを実装するときのレイヤー構成と書く順番は？

機能APIは Controller → Service → Repository → Prisma の依存方向で構成。書く順番は 下層からのボトムアップ が推奨。

依存方向

Controller ← HTTPの入口 ↓ 依存 Service ← ビジネスロジック ↓ 依存 Repository ← Prismaへのアクセス集約 ↓ 依存 PrismaClient ← DB

書く順番（ボトムアップ）

  1. DTO: 入出力の型契約を最初に決める
  2. Repository: Prismaを叩く層
  3. Service: Repositoryを呼ぶビジネスロジック
  4. Controller: HTTPエンドポイントを定義してServiceを呼ぶ
  5. Module: providers / controllers でDI配線
  6. AppModule登録: imports に追加して有効化

なぜボトムアップ？

下層から書く方が 「使いたい関数がすでに存在している状態」で上層を書けるので、IDEの型補完が効きやすい。逆のトップダウンだと、Controllerを書いてる時点で this.service.list() の戻り値型が未定で補完が利かず辛い。

各層に書くこと（実装時の視点）

層具体的に書くもの
DTOcreateZodDto(getCategoriesQuerySchema) を継承したクラス。Zodスキーマは shared に置く
RepositoryfindAll() 等のメソッド。where: { deletedAt: null } の論理削除フィルタもここ
ServiceRepositoryを呼ぶメソッド。複数Repositoryをまたぐ集約処理もここ
Controller@Get() 等のデコレータ + ServiceメソッドのIN/OUTを変換するだけ
Moduleproviders: [Service, Repository], controllers: [Controller]

Category APIの例

// 1. DTO（dto/getCategories.dto.ts）
export class GetCategoriesQueryDto extends createZodDto(getCategoriesQuerySchema) {}

// 2. Repository（categories.repository.ts）
@Injectable()
export class CategoriesRepository {
  constructor(private readonly prisma: PrismaService) {}
  async findAll(type?: TransactionType) {
    return this.prisma.category.findMany({
      where: { deletedAt: null, ...(type && { type }) },
    });
  }
}

// 3. Service（categories.service.ts）
@Injectable()
export class CategoriesService {
  constructor(private readonly repo: CategoriesRepository) {}
  list(type?: TransactionType) { return this.repo.findAll(type); }
}

// 4. Controller（categories.controller.ts）
@Controller('categories')
export class CategoriesController {
  constructor(private readonly service: CategoriesService) {}
  @Get()
  list(@Query() query: GetCategoriesQueryDto) {
    return this.service.list(query.type);
  }
}

// 5. Module（categories.module.ts）
@Module({
  controllers: [CategoriesController],
  providers: [CategoriesService, CategoriesRepository],
})
export class CategoriesModule {}

// 6. AppModule に登録
@Module({ imports: [CategoriesModule, ...] })
export class AppModule {}
覚え方: 「型契約を決める → 下から積む → 配線」。書き始める前にDTO（＝API契約）を決めると迷いが減る。
DTO + createZodDto + nestjs-zod でバリデーションはどう動く？

DTOクラスの正体は 「Zodスキーマを抱えた箱」。createZodDto() がスキーマをクラスに包み、NestJSの ZodValidationPipe がそれを認識して自動でバリデーションする。

仕組みの流れ

GetCategoriesQueryDto (クラス) └─ createZodDto(getCategoriesQuerySchema) を継承 └─ 内部にZodスキーマを保持 └─ NestJSの ZodValidationPipe がこのクラスを認識 └─ @Query() で受け取る瞬間に検証実行 └─ 失敗 → ZodValidationException └─ 成功 → 型付きの値が Controller に渡る

コード例

// packages/shared/src/schemas/category.ts
export const getCategoriesQuerySchema = z.object({
  type: transactionTypeSchema.optional(),
});

// apps/backend/src/modules/categories/dto/getCategories.dto.ts
import { createZodDto } from 'nestjs-zod';
import { getCategoriesQuerySchema } from '@pick-cash/shared/schemas/category.ts';

export class GetCategoriesQueryDto extends createZodDto(getCategoriesQuerySchema) {}

// apps/backend/src/main.ts
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ZodValidationPipe());  // ← 1行入れるだけで全エンドポイント自動検証
  await app.listen(3000);
}

失敗時のフロー

?type=invalid のような不正値が来たとき:

クライアント: GET /categories?type=invalid ↓ ZodValidationPipe: schema.parse() → 失敗 ↓ ZodValidationException がthrowされる ↓ GlobalExceptionFilter で捕捉 ↓ レスポンス: { statusCode: 400, message: 'Bad Request' } ↓ 詳細はサーバの console.error に出力（情報漏洩しない）

main.ts 1行で全エンドポイントに効く理由

useGlobalPipes() はパイプラインの Pipeレイヤー全体に1つのPipeを適用 する。各Controllerメソッドの引数デコレータ（@Body(), @Query() など）が値を受け取る直前に必ず通る。よってDTOクラスを書いておくだけで自動的に検証が走る。

覚え方: DTOは「箱」、Zodスキーマは「中身のルール」、ZodValidationPipeは「全エンドポイントに自動で適用される検査官」。3点セットで 「リクエストが入る瞬間に必ず検査される」状態が作れる。

環境構築ハマりポイント

Node・TypeScript・モノレポで実際に遭遇したハマりとその回避方法。

ESM と CJS の共存について（NestJS + pnpm workspace 構成）

pick-cashは 「ソース上はESM風、コンパイル後はCJS、共有パッケージはESM風TS」という3層キメラ状態。これがハマりの根本原因になる。

前提: ESM vs CJS

CJS（CommonJS）ESM（ES Modules）
構文const x = require('./foo') / module.exports = ...import x from './foo' / export ...
拡張子の厳しさ緩い（./foo で foo.js や foo/index.js を自動解決）厳しい（./foo.js のように拡張子必須）
由来Node.js が最初に採用したモジュール仕様ECMAScript 標準のモジュール仕様
動的読込同期非同期（import()）

pick-cashの実態

  • ソースは全部 import / export 構文（ESM風の見た目）
  • backend の tsconfig.json が "module": "commonjs" なので、コンパイル後は require に変換される
  • packages/shared はビルドしない（TSのまま読まれる）
  • 結果、「backend(CJS) ↔ shared(ESM風TS)」の境界で変換が発生

なぜNestJSはCJSがデフォルトなのか

  • reflect-metadata の依存: ESM環境で動かすには初期化順序の調整が必要
  • DI初期化順序: ESMのトップレベルawaitと相性が悪い
  • 公式CLIテンプレ: nest new がCJSで作る

近年は対応しつつあるが、現状のスムーズな解はCJSのまま。

ビルド後の実物（概念）

// 書いたコード（main.ts）
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

// コンパイル後（dist/main.js）
"use strict";
const core_1 = require("@nestjs/core");
const app_module_1 = require("./app.module");

3層キメラの図

[ソース層] backend/src/*.ts ← ESM風 import/export ↓ tsc / swc [ビルド出力層] backend/dist/*.js ← CJS require ↓ require('@pick-cash/shared') [共有パッケージ層] packages/shared/src/*.ts ← ESM風TS（ビルドしない）
ここがハマる: backendはCJSで動くが、共有パッケージはTSのまま読み込まれるためNodeのモジュール解決ルールが ESM寄りに切り替わるケースがある。次の Q を参照。
Node v24 で shared パッケージから ERR_MODULE_NOT_FOUND が出る問題

backendのビルドやテストでは出ないが、実機起動（pnpm dev:backend）でだけ shared 経由のimportが ERR_MODULE_NOT_FOUND で落ちる。

原因

Node v24 が CJSからTSをrequireする経路でESM resolverを動かす挙動になった。ESM resolverは拡張子必須なので、shared/index.ts内の export * from './schemas/category'（拡張子なし）が解決できずに落ちる。

フロー図

[backend実行時] require('@pick-cash/shared') → packages/shared/src/index.ts （ESM風TS） ├─ export * from './schemas/category' ← 拡張子なし │ → Node v24 の ESM resolver が動く │ → .ts 拡張子なし → ERR_MODULE_NOT_FOUND │ └─ export * from './schemas/transaction' ← 同じく失敗

解決策3案

案内容採用
Aimportに .ts 拡張子を明示
export * from './schemas/category.ts'		採用
Bshared/package.json に "type": "module" を追加影響範囲が大きい
Csharedをビルドして dist 経由で配布学習プロジェクトでは過剰

採用した解（A）

// packages/shared/src/index.ts
export * from './schemas/category.ts';     // ← .ts を付ける
export * from './schemas/transaction.ts';

VSCodeのエラーは別問題

上の対応をすると、VSCodeで 「allowImportingTsExtensions が有効である場合〜」というエラーが出る。これはTypeScript本体（tsserver）が出しているエラーで、ビルド・実行には影響しない。詳細は次のQへ。

覚え方: 「ビルドは通るのに実行で落ちる」「テストは通るのに pnpm dev で落ちる」 → モジュール解決の経路差異を疑う。Node のバージョンとESM resolverの挙動を最初に確認。
allowImportingTsExtensions とは？いつ使う？

TypeScriptのコンパイラオプション。import './foo.ts' のように .ts 拡張子を含んだimportを許可する設定。デフォルトでは禁止されている。

なぜデフォルトで禁止？

TypeScriptの基本思想は「.ts はコンパイル後に消えるべき」。foo.ts をimportしても、コンパイル後の foo.js ではパスが噛み合わない。だからデフォルトでは ./foo や ./foo.js を書くのが正規ルート。

経路ごとの挙動の違い

経路tsconfigを参照するか.ts拡張子付きimportの扱い
VSCode の tsserverする（shared/tsconfig.json を厳格に適用）allowImportingTsExtensions: false ならエラー表示
backend ビルド（swc経由）tsserverほど厳しくないswcはts拡張子をエラー判定しない
Node ランタイム無関係（tsconfigを読まない）ファイルパスで解決できればOK

採用した設定

// packages/shared/tsconfig.json
{
  "compilerOptions": {
    "allowImportingTsExtensions": true,
    "noEmit": true   // ← 必須セット
  }
}

noEmit: true とセットなのはなぜ？

TypeScriptの論理: 「.ts 付きでimportを書く＝コンパイル後のJSにそのまま出るとマズい → TypeScriptが出力しない（noEmit）なら矛盾しないからOK」。「TSがJSを吐かないなら、ソース上に.tsが残っていても問題ない」という論理。

[許可される組み合わせ] allowImportingTsExtensions: true ＋ noEmit: true ← セット必須 ＝ TSはJSを出力しない、ソースのまま使う [禁止される組み合わせ] allowImportingTsExtensions: true ＋ noEmit: false ← エラー ＝ JSを吐くのに.ts付きimportが残るので破綻
pick-cashでの位置づけ: sharedパッケージは「ビルドせずTSのまま使う」方針なので noEmit: true。よって allowImportingTsExtensions: true も付けられる。これで Node v24 の ESM resolver の拡張子必須要求も満たせる、というのが 3層キメラ問題の最終解。
