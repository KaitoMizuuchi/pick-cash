# save-knowledge

学習プロジェクトで Claude が説明した内容を `docs/knowledge/index.html` に蓄積するためのスキル。

## いつ起動するか

ユーザーが次のような発言をしたとき:

- 「これナレッジとして保存しといて」
- 「学習した内容を残しておきたい」
- 「ナレッジ化して」
- 「今の内容を記録に残して」
- 「learning log に追加して」
- 「あとから見返せるようにまとめておいて」

明示的に「保存」「記録」「残す」「ナレッジ」のような語が出たら、説明系のやりとりが直前にあったかを確認した上で起動する。学習プロジェクトの性質上、過去のチャットで丁寧に説明したトピックは資産。逃さず保存することが重要。

## 何をするか

1. **既存ナレッジファイル `docs/knowledge/index.html` の構造を把握する**（カテゴリ・既存Q&A・スタイル）
2. **直近の会話から「学習トピック」を抽出する**（疑問→解説のペアになっているもの）
3. **既存カテゴリに追加するか、新規カテゴリを作るかを判断**
4. **HTMLを追記する**（既存スタイル・粒度を踏襲）
5. **目次（TOC）を更新**
6. **追記した内容のサマリをユーザーに報告**

## 重要な原則

### 粒度を圧縮しない

学習プロジェクトの目的は「後から見返して理解を再現すること」。Q&Aの解説部分は **チャットで実際に説明された粒度をそのまま保つ** こと。具体的には:

- コード例・図（ASCIIアート）・比較表は省略しない
- 「なぜ」を含む説明は丸ごと残す
- 「ユーザーが疑問を持って、その回答を読んで腑に落ちた」プロセスを HTML 上で再現できる粒度

「概要だけ短く書く」は禁止。それなら memo に書けば済む。HTML化する意味は、図表込みで丁寧に残すこと。

### 既存スタイルを徹底踏襲する

`docs/knowledge/index.html` には独自のスタイルガイド（CSS変数、アコーディオン構造、`.ascii-art`/`.note`/`.badge` などのクラス）がある。**新規にスタイルを発明しない**。既存のHTMLを必ず読み、同じパターンで追記する。

### 既存カテゴリへの追加を優先する

新しい疑問が既存カテゴリ（例: "@デコレータ"）に関連するなら、新規カテゴリを作らずそのカテゴリ内に Q&A を追加する。新規カテゴリは「明確に異なるトピック」のときだけ作る。

## 実行手順

### Step 1: 既存ファイルの構造を把握

`docs/knowledge/index.html` を読んで以下を把握する:

- カテゴリ一覧（`<section class="category" id="...">`）
- 各カテゴリの既存 Q&A の見出し
- 使われているHTML要素・CSSクラス（`details.qa` / `summary` / `.qa-body` / `.ascii-art` / `pre` / `code` / `table` / `.note` / `.badge`）
- TOC（`<nav class="toc">` 内の `<ul>`）

### Step 2: 会話から学習トピックを抽出

直近の会話履歴を遡り、以下のパターンを探す:

- ユーザーの **質問・疑問** が起点になっているやりとり
- それに対する Claude の **解説** が一定の長さ・密度を持っているもの
- 概念整理・フレームワーク仕様・設計判断の理由など、**コードを読めば自明ではない知識**

候補として複数浮かんだら、Q&A 単位で整理する。1つの大きな質問に複数の論点があれば、論点ごとに別 Q&A に分けてもよい（既存ファイル内で `@Module / @Injectable / @Controller / @Catch の違いは？` のような統合質問になっているケースもあるので、ケースバイケース判断）。

### Step 3: 既存カテゴリ vs 新規カテゴリの判定

- 既存カテゴリの主題と関連が深ければ、そのカテゴリの `<section>` 末尾に Q&A を追加
- 既存カテゴリにフィットしないなら新規カテゴリを作る
  - `<section class="category" id="...">` を `<main>` 末尾（最後の `</section>` の直後）に追加
  - TOC の `<ul>` にも対応する `<li><a href="#...">...</a></li>` を追加

### Step 4: HTML 追記

既存のスタイルパターンを踏襲して `<details class="qa">` ブロックを構築する。

#### 基本テンプレ

```html
<details class="qa">
  <summary>ここに疑問文（〜は？／〜って何？／〜とどう違う？）</summary>
  <div class="qa-body">
    <p>導入の一文。</p>

    <h4>セクション見出し1</h4>
    <p>解説本文。</p>

    <pre><code>// コード例</code></pre>

    <h4>比較表</h4>
    <table>
      <thead><tr><th>項目</th><th>説明</th></tr></thead>
      <tbody>
        <tr><td>A</td><td>...</td></tr>
      </tbody>
    </table>

    <div class="ascii-art">図やツリー構造</div>

    <div class="note">
      <strong>覚え方:</strong> ポイントの要約。
    </div>
  </div>
</details>
```

#### 使えるクラス

| クラス | 用途 |
|---|---|
| `.qa` | Q&A アコーディオン（`<details>` に付ける） |
| `.qa-body` | 解説本文ラッパー（`<div>` に付ける） |
| `.ascii-art` | ASCIIアート / ツリー図 |
| `.note` | 「覚え方」「まとめ」「補足」などのボックス |
| `.note.warn` | 警告系の補足 |
| `.badge` | 緑色のバッジ（採用・通常） |
| `.badge.good` | 緑バッジ（推奨） |
| `.badge.warn` | 黄バッジ |
| `.badge.bad` | 赤バッジ |
| `.check` | 緑のチェック記号スタイル |
| `.cross` | 赤のバツ記号スタイル |

#### HTMLエスケープ

`<` / `>` / `&` がコード例に含まれる場合、適切にエスケープ:
- `<` → `&lt;`
- `>` → `&gt;`
- `&` → `&amp;`

特に TypeScript の Generics（`Promise<T>`）は `Promise&lt;T&gt;` と書く。

### Step 5: TOC 更新（新規カテゴリ作成時のみ）

`<nav class="toc"><ul>` の中に `<li><a href="#新カテゴリID">カテゴリ名</a></li>` を追加。

### Step 6: 報告

ユーザーに以下を伝える:

- 追加した Q&A の数とタイトル
- 追加先カテゴリ（既存 or 新規）
- 確認用URL（例: `http://localhost:8000/index.html` で確認可能）

## サブエージェントで実行する

このスキルが起動したら、メインコンテキストを節約するため **`general-purpose` サブエージェント** に作業を委譲する。サブエージェントには以下を渡す:

- 対象ファイルパス: `docs/knowledge/index.html`
- このスキルファイルへのパス: `.claude/skills/save-knowledge/SKILL.md`
- 直近の会話で抽出すべき学習トピックの **要点と該当ターン** （メインセッションが整理して渡す）
- 「既存スタイルを踏襲して追記。粒度は圧縮しない。HTML エスケープ忘れない。」という念押し

### サブエージェント呼び出しテンプレ

```
description: "学習ナレッジを index.html に追加"
subagent_type: "general-purpose"
prompt: |
  pick-cash プロジェクトの学習ナレッジ蓄積タスク。

  ## 対象ファイル
  /home/kaitomizuuchi/practice/pick-cash/docs/knowledge/index.html

  ## スキル定義
  詳細手順は /home/kaitomizuuchi/practice/pick-cash/.claude/skills/save-knowledge/SKILL.md
  必ず最初に読むこと。

  ## 追加するトピック
  - トピック1: 「<疑問文>」
    要点: <要約>
    粒度の指針: <チャットで触れた具体例・図・表・コード断片など>
  - トピック2: ...

  ## やってほしいこと
  1. 既存 index.html を Read して構造・スタイルを把握
  2. SKILL.md の手順に従い、既存カテゴリへの追加 / 新規カテゴリ作成を判断
  3. <details class="qa">...</details> ブロックを既存パターン通りに作成
  4. ファイル末尾ではなく、該当カテゴリの最後の Q&A の直後に追加
  5. 新規カテゴリの場合は <main> 末尾と TOC <ul> を更新
  6. 完了後、追加した Q&A のタイトル一覧と追加先カテゴリを報告

  ## 重要な制約
  - チャットの粒度を圧縮しない（コード例・比較表・ASCII図は省略禁止）
  - 既存の CSS クラスのみ使う（新規スタイル発明禁止）
  - HTMLエスケープ忘れない（<, >, & は &lt; &gt; &amp;）
```

## やってはいけないこと

- 学習トピックを「概要だけ」に圧縮する（粒度を保つこと）
- 既存スタイルを無視して新しいクラスや色を追加する
- 同じトピックの重複追記（既存Q&Aを確認）
- 別ファイルを新規作成する（このスキルは `docs/knowledge/index.html` 一本に集約）
- ユーザー確認なしに既存Q&Aを書き換える（追記のみ、書き換えが必要な場合は事前確認）

## 補足

- cSpell の警告（"Usecase", "IDDD" などの英単語警告）は学習用語として残してOK。無視して進めてよい
- 追加した内容が既存と被ったら、新規追加せず既存を更新する判断もあり（その場合はユーザーに確認）
- WSL環境でHTMLを確認するには `python3 -m http.server 8000` を `docs/knowledge/` で実行 → `http://localhost:8000/index.html`
